Deep Linux for DevOps, Cloud & Platform Engineering
A 10-week mental-model-driven plan (WSL-based, 2-4 hrs/day)
________________________________________
How this plan is different from a tutorial list
Every week teaches one mental model of how Linux actually behaves, not a bucket of commands. Commands appear as the natural output of understanding the model — the same way knowing grammar lets you form sentences you've never memorized. Each day ends with a DevOps/Cloud/Platform task that forces you to use the model, because a model you can recite but not use isn't learned yet.
Two rules to follow all 10 weeks:
1.	Never copy-paste a command without predicting what it will do first. Guess, run it, compare. This single habit is what builds intuition instead of memorization.
2.	Break something every single day. Delete a config, kill the wrong process, fill the disk, lock yourself out of sudo. Then fix it. You will remember the fix forever; you will forget a tutorial by Friday.

Week 1 — Linux Is Just Files (the filesystem mental model)
The model: Unix's defining idea is "everything is a file" — devices, processes, sockets, even kernel settings are exposed as files you can read/write with the same tools. Once this clicks, /proc, /dev, /sys stop being mysterious and become "oh, it's just a file."
Day	Focus	Hands-on DevOps task
1	Filesystem Hierarchy Standard: why /etc, /var, /usr, /opt, /bin vs /usr/bin exist (historical reasons + current convention). Tour your WSL filesystem with tree/ls -la recursively.	Find where Docker stores its data on disk (/var/lib/docker) and explain why it's under /var and not /opt.
2	Absolute vs relative paths, ./.., symlinks vs hard links (and why symlinks matter for versioned binaries like multiple Node/Python versions).	Create a symlink scenario: install two "versions" of a script, symlink current to one, switch it — this is literally how tools like nvm/pyenv and blue-green binary swaps work.
3	File types beyond regular files: directories, symlinks, device files, sockets, pipes. file and stat commands; inode concept (a file's identity is its inode, not its name).	Use stat to prove that a hardlinked file and its "original" are literally the same inode — relevant for understanding how Docker layers and bind mounts behave.
4	/proc and /sys as "the kernel exposed as files." Read /proc/cpuinfo, /proc/meminfo, /proc/<pid>/status.	Find a running process's open file descriptors via /proc/<pid>/fd — this is exactly what you're debugging when a container "won't release a file handle."
5	Permissions properly: user/group/other, the octal system (why 755 means what it means, derive it instead of memorizing), chmod/chown/chgrp, setuid/setgid/sticky bit conceptually.	Reproduce a real bug: create a file only root can write, have a "non-root" simulated process try to write to it, fix via correct ownership instead of chmod 777 (explain why 777 is the wrong instinct).
6	Hard vs soft links to mounted filesystems, disk usage tools du/df, understanding mount points conceptually (foreshadow Docker volumes).	Fill your WSL disk artificially with dd or a loop, observe df change, diagnose and clean with `du -sh */
7	Review + synthesis day. No new material. Re-derive the FHS from memory, explain permissions to a rubber duck (or me) out loud.	Write a one-page personal cheat-sheet in your own words — not copied — of what you learned. This forces retrieval, which is what actually cements memory.
Week 1 checkpoint: You should be able to explain why a Dockerfile's COPY and VOLUME instructions interact with the filesystem the way they do, using inode/mount concepts, not just "it copies files."

Week 2 — Linux Boots and Organizes Itself (init, systemd, the kernel's job)
The model: A running Linux machine is a kernel managing hardware + a tree of processes started by systemd (PID 1) according to declared "units." Every server you'll ever SSH into for a DevOps job follows this exact model.
Day	Focus	Hands-on task
8	The boot sequence conceptually (BIOS/UEFI → bootloader → kernel → initramfs → systemd) — WSL2 skips most of this (it has its own init, init in WSL is partially systemd now), so treat this as theory you'll recognize on real VMs/cloud instances later.	Compare `ps -ef
9	systemd units: services, targets, timers. systemctl status/start/stop/enable/disable. Why enabled ≠ running.	Write your own tiny systemd service file that runs a script on a timer (a poor-man's cron) — this is precisely the skill behind deploying any background worker on a cloud VM.
10	Logs: journalctl deeply — filtering by unit, by time, by priority, following live (-f), why systemd centralized logging replaced scattered /var/log files (and /var/log still matters for non-systemd-managed logs).	Intentionally make your custom service fail (bad path, missing permission) and diagnose purely from journalctl -u yourservice -xe. This is the single most common real-world DevOps debugging motion.
11	Runlevels/targets (multi-user.target, graphical.target etc conceptually), dependency ordering between units (Wants=, Requires=, After=).	Make a second systemd service that must start after your Day 9 service and depends on it — observe what happens if you stop the dependency.
12	Package management as "how software gets onto the filesystem properly" — apt, dpkg, what a .deb actually contains, repository concept.	Install a package, then manually inspect what files it placed where with dpkg -L <package> — connect this back to Week 1's FHS knowledge.
13	Cron vs systemd timers in depth — when DevOps pipelines still use cron (simplicity, portability) vs timers (logging, dependency awareness).	Build the same scheduled task two ways (cron and systemd timer), compare debugging experience.
14	Review + synthesis. Re-derive boot → systemd → service lifecycle out loud without notes.	Diagram (on paper or in chat with me) the full lifecycle of "I deploy a new background worker on a cloud VM" using only this week's concepts.
Week 2 checkpoint: Given a broken/unknown systemd service on any Linux box, you should be able to diagnose it within a few minutes using systemctl + journalctl alone.

Week 3 — Processes Are the Living System (process & resource model)
The model: Every running thing is a process with a PID, a parent, an owner, signals it can receive, and resource usage. Kubernetes pods, Docker containers, and cloud VM workloads are all, underneath, just processes with extra isolation layered on (this is the seed for Week 9-10).
Day	Focus	Hands-on task
15	Process tree: ps, pstree, parent/child relationships, what happens to children when a parent dies (orphans, zombies conceptually).	Spawn a background process, kill its parent, observe orphaning with pstree.
16	Signals properly: SIGTERM vs SIGKILL vs SIGHUP vs SIGINT — why graceful shutdown matters (this directly explains Kubernetes pod termination grace periods and Docker stop timeouts).	Write a script that traps SIGTERM and cleans up before exiting, then kill it with SIGTERM vs SIGKILL and compare behavior — this is exactly the logic behind a well-behaved containerized app.
17	Foreground/background jobs, &, nohup, disown, jobs, fg/bg, and why this matters less once you understand systemd/containers manage long-running processes properly.	Start a long task, background it, disown it, close the terminal, reconnect and confirm it survived — then do it "properly" with systemd instead and compare reliability.
18	Resource monitoring: top/htop, load average (what the three numbers actually mean mathematically), memory (used vs cached vs available — the classic "Linux ate my RAM" misunderstanding).	Generate CPU load deliberately (yes > /dev/null & a few times), watch load average climb, kill it off, watch it settle — directly relevant to reading cloud monitoring dashboards correctly.
19	cgroups and namespaces conceptually (just enough — full depth comes in Week 9 with Docker). This is the kernel feature that makes containers possible at all.	Look at /sys/fs/cgroup on your WSL system, find a process's cgroup, connect it mentally to "this is how Docker limits a container's memory."
20	Process priority (nice/renice), ulimit and resource limits — why a "noisy neighbor" process matters on shared infrastructure.	Set a ulimit for max open files, hit the limit deliberately with a script, observe the failure — this is the exact failure mode behind real "too many open files" production incidents.
21	Review + synthesis.	Explain, without notes, what actually happens (in process terms) when you run docker stop on a container.
Week 3 checkpoint: You can explain Kubernetes' graceful pod termination and Docker's container lifecycle purely in terms of signals and process trees — no hand-waving.

Week 4 — Storage and Devices (how Linux sees disks, and why it matters for volumes)
Day	Focus	Hands-on task
22	Block devices vs character devices, /dev tour, partitions conceptually, lsblk/blkid.	Inspect your WSL virtual disk's structure with lsblk and connect it to how cloud EBS/managed disks are just block devices attached to a VM.
23	Filesystems (ext4 conceptually, what formatting actually does), mount/umount, /etc/fstab syntax and purpose.	Create a loopback file, format it as ext4, mount it, write to it, unmount — this is precisely the mental model behind a Docker volume or a cloud persistent disk.
24	Disk usage forensics: du, df, finding what's actually eating space, inode exhaustion (a real and confusing failure mode: disk has space but "no space left on device").	Deliberately exhaust inodes (create thousands of tiny files) and diagnose the difference between "out of space" and "out of inodes."
25	Swap, conceptually — what it is, why cloud VMs sometimes disable it, OOM killer behavior.	Read your WSL's current swap config, explain in your own words when the OOM killer activates and why that matters for sizing a Kubernetes pod's memory limits.
26	Archiving and compression as a storage/transport topic: tar, gzip, why .tar.gz is two steps not one, when to use zip instead.	Package a directory the "DevOps way" (tar czf), then unpack and verify integrity — same operation as preparing a build artifact for deployment.
27	Review + synthesis.	Explain end-to-end what happens, in storage terms, when a Kubernetes PersistentVolumeClaim gets bound — even just conceptually, tying back to mount/block device knowledge.
28	Buffer/catch-up day. Revisit anything shaky from Weeks 1-4 before moving to networking.	Free choice: re-do whichever hands-on task felt weakest.

Week 5 — Linux Talks to the Network (the networking model)
The model: Networking on Linux is sockets, interfaces, routes, and ports — and almost every cloud networking concept (VPCs, security groups, load balancers) is a managed abstraction over these same primitives.
Day	Focus	Hands-on task
29	Network interfaces, IP addressing basics, ip a, ip route (replacing legacy ifconfig/route mentally so you recognize both).	Map your WSL's network interface and default route, explain in your own words how WSL2's NAT networking actually works underneath.
30	DNS resolution: /etc/resolv.conf, /etc/hosts, dig/nslookup, how a hostname becomes an IP step by step.	Add a fake entry to /etc/hosts, curl it, observe resolution bypassing real DNS — this is exactly how local dev environments fake production domains.
31	TCP/UDP ports and sockets conceptually, ss -tulpn to see what's listening, connecting "a service is down" to "nothing is listening on that port."	Start a simple Python/Node HTTP server on a port, confirm with ss, kill it, confirm the port frees up — directly the skill used to debug "port already in use" errors.
32	curl/wget deeply — headers, status codes, following redirects, talking to APIs from the shell (this is core DevOps glue work).	Hit a real public API with curl, inspect headers with -i, handle a non-200 response, pipe JSON into jq (small preview, formalized later).
33	Firewalls conceptually: iptables/nftables basics or ufw if simpler, and crucially how this maps to cloud security groups/NSGs (same idea, different abstraction layer).	Block a port with ufw/iptables locally, confirm the block with curl/ss, then unblock — narrate out loud how this maps to "why can't my service reach the DB" cloud tickets.
34	SSH properly: key-based auth mechanics (not just "run ssh-keygen"), ~/.ssh/config for managing multiple hosts, agent forwarding conceptually.	Generate a keypair, set up ~/.ssh/config with an alias for a host, SSH into your own WSL via localhost to prove the full loop works.
35	Review + synthesis.	Explain, end to end, what happens at the OS level when a load balancer health-checks a backend instance — interfaces, ports, sockets, response codes, all of it.
Week 5 checkpoint: Cloud networking diagrams (VPC/subnet/security group/load balancer) should now look like "named, managed versions of things I already understand at the Linux level."

Week 6 — Users, Identity, and Security (who can do what, and why)
Day	Focus	Hands-on task
36	Users/groups deeply: /etc/passwd, /etc/group, /etc/shadow structure, UID/GID, why root is UID 0 specifically (not just "magic").	Create a new user and group by hand (not via useradd, by editing the files directly with care, in a disposable WSL snapshot if possible) to truly understand what the helper commands automate.
37	sudo properly: /etc/sudoers, principle of least privilege, why "just sudo everything" is a real production risk (and connects directly to IAM thinking in cloud).	Configure a limited sudo rule (a user can only restart one specific service, nothing else) — this is the exact mental model behind cloud IAM policies.
38	SSH key management at scale conceptually — authorized_keys, why this doesn't scale past a handful of servers (motivating config management tools like Ansible later).	Add a second key to authorized_keys, test login with both, remove one, confirm revocation — this is "access management" in its rawest form.
39	File permission edge cases revisited with security lens: world-writable directories, the sticky bit on /tmp explained properly now that you understand processes/users together.	Find any world-writable files on your system with find / -perm -002 -type f 2>/dev/null (read-only exploration), explain why each either matters or doesn't.
40	Secrets handling on Linux conceptually — why plaintext secrets in files/env vars are risky, environment variable exposure via /proc/<pid>/environ, motivating vaults/secrets managers used in real DevOps stacks.	Demonstrate the leak yourself: export a fake secret as an env var, read it back via /proc/<pid>/environ of that process — a genuinely eye-opening exercise.
41	Auditing basics: last, who, w, log locations for auth (/var/log/auth.log or via journalctl), just enough to know where to look during an incident.	Review your own WSL's login history and explain what you're seeing.
42	Review + synthesis.	Write (in your own words) a short "security model" paragraph connecting users/permissions/sudo/SSH into one coherent story — this is genuinely interview-relevant content.

Week 7 — Commanding Linux: The Shell as a Real Tool (not isolated commands)
The model: The shell is a programming environment, not a list of commands to memorize. Once you can write a script, you stop "looking things up" and start "composing tools."
Day	Focus	Hands-on task
43	Shell fundamentals properly: variables, quoting rules (the actual reason "$var" vs $var matters), command substitution, exit codes and $?.	Write a 10-line script that checks if a service is running and prints a clear message based on the exit code — foundational to every CI/CD health check script you'll ever write.
44	Pipes and redirection as composition, not memorized symbols: build a mental model of stdin/stdout/stderr as just file descriptors 0/1/2 (ties back to Week 1's "everything is a file").	Build a pipeline of 4+ commands solving a real problem (e.g., "find the 5 largest log files modified in the last day") — verify each stage by running partial pipelines first.
45	Text processing trio: grep, sed, awk — deep enough to be dangerous. Regex basics, in-place editing, field-based processing.	Parse a real (or sample) nginx/app log file: extract all 5xx errors, count by IP, sort by frequency — this exact skill is used constantly in on-call debugging.
46	Loops and conditionals in bash properly, functions, reading input, argument parsing ($1, $@, getopts basics).	Write a reusable script with flags (e.g., ./backup.sh --source /path --dest /path2) — this is the shape of every internal tooling script in a DevOps team.
47	jq for JSON (since virtually every cloud CLI and API returns JSON), combining with curl from Week 5.	Pull a real API response, extract specific nested fields with jq, reformat into a clean table — directly the skill used to parse aws cli/kubectl -o json output.
48	Idempotency and safety in scripts — why DevOps scripts must be safe to re-run, set -euo pipefail and why each flag exists, trapping errors.	Take your Day 46 script and harden it: add set -euo pipefail, error handling, and make it safe to run twice without side effects.
49	Review + synthesis.	Combine the week into one capstone script: monitors a log file, extracts errors with grep/awk, summarizes with jq-style structure, and emails/prints an alert — fully your own design.
Week 7 checkpoint: Given almost any text-processing or automation problem, you should reach for composing shell tools rather than searching "how to do X in bash."

Week 8 — Configuration, Automation, and "Infrastructure-as-Text" Thinking
Day	Focus	Hands-on task
50	Environment variables and config precedence (system-wide vs user vs shell session vs process-level) — why this matters enormously for containerized apps and CI/CD pipelines.	Demonstrate the full precedence chain yourself by overriding the same variable at each level and proving which one wins.
51	Config file formats common in DevOps tooling (YAML/INI/conf-style) and how Linux services typically consume them — read a real nginx.conf or systemd unit file structurally.	Write a minimal nginx config from scratch (install nginx in WSL if not present) that proxies to your Day 31 test server — connects networking + config knowledge together.
52	Version control hygiene for "infra-as-text": why .gitignore matters for secrets, basic git workflow refresher if needed (skip if already solid) — this is glue knowledge for any IaC work ahead.	If git is already solid, skip to: set up a small repo containing your Week 7 scripts properly structured with a README — practicing the habit of treating ops scripts as real code.
53	Package/dependency management mental model applied to scripting (e.g., pinning versions, why "works on my machine" happens) — sets up Docker week conceptually.	Reproduce a "works on my machine" failure deliberately (script depends on a tool version), then fix it by being explicit about dependencies — foreshadows why Docker exists.
54	Cross-distro awareness: enough Alpine/RHEL/Debian differences to not be surprised (apk vs apt vs yum/dnf, musl vs glibc briefly) — critical since Docker base images vary wildly.	Pull an Alpine-based container, notice missing tools compared to Ubuntu, install what's needed — directly prepares you for Dockerfile decisions next week.
55	Logging conventions and log rotation (logrotate basics) — why uncontrolled logs fill disks in production.	Configure a basic logrotate rule for a log file you generate yourself, force rotation, confirm it worked.
56	Review + synthesis.	Write a short doc (for yourself) titled "How I'd configure a fresh Linux server for a small app" using only Weeks 1-8 knowledge — no external tools yet.

Week 9 — Containers Are Just Linux Wearing a Costume
The model: Docker is namespaces + cgroups + a union filesystem, all concepts you already have from Weeks 1, 3, and 4. This week makes that explicit so containers stop feeling like magic.
Day	Focus	Hands-on task
57	Revisit namespaces and cgroups (Week 3, Day 19) at full depth now: PID namespace, network namespace, mount namespace — explicitly mapped to what docker run does.	Run a container, then from the host inspect its process via ps aux — find its PID on the host vs PID 1 inside the container, proving namespaces are "just a view," not separate machines.
58	Docker images as layered filesystems — connect directly to Week 1's inode/filesystem knowledge. docker history, image layers, why layer caching works.	Build two versions of a Dockerfile (good layer ordering vs bad) and compare build cache behavior and image sizes.
59	Dockerfile as a shell script with structure — every instruction maps to something you already know (RUN = shell, COPY = filesystem, USER = Week 6 identity, EXPOSE = Week 5 ports).	Write a Dockerfile from scratch (no copying a template) for one of your Week 7 scripts, applying least-privilege (non-root USER) from Week 6.
60	Container networking: bridge networks, how container-to-container communication maps to the Week 5 networking model (it's just network namespaces + virtual interfaces + NAT).	Run two containers on a custom Docker network, get them talking via container name resolution, and explain why that resolution works (Docker's embedded DNS).
61	Volumes and bind mounts properly — directly Week 4's mount knowledge, now applied. Persistent data vs ephemeral container filesystem.	Demonstrate data loss without a volume (delete container, data gone) vs persistence with a volume — narrate the mount mechanics underneath.
62	Resource limits on containers (--memory, --cpus) — directly Week 3's cgroups knowledge, now with hands on the exact controls.	Set a memory limit below what a test process needs, watch the OOM killer (Week 4) terminate it inside the container, read the exit code/logs to confirm why.
63	Review + synthesis.	Explain to yourself (or me) the full lifecycle of docker run purely in terms of namespaces, cgroups, and filesystem layers — no Docker-specific jargon allowed in the explanation.
Week 9 checkpoint: Containers should now feel like "Linux process isolation with good tooling," not a separate technology to learn from scratch.

Week 10 — Orchestration & Cloud Are Linux at Scale
The model: Kubernetes nodes are Linux machines running container runtimes; cloud VMs are Linux (or Windows, but you care about Linux) machines with managed networking/storage layered on. This week is pure synthesis — minimal new Linux concepts, maximum connecting-the-dots.
Day	Focus	Hands-on task
64	Local Kubernetes setup (k3s, minikube, or kind — pick the lightest for WSL) purely as "many of Week 9's containers, coordinated."	Get a single-node local cluster running, kubectl exec into a pod, and from there run the exact Linux diagnostic commands from Weeks 1-3 (ps, df, ip a) inside the pod — proving it's just Linux.
65	Pod lifecycle mapped explicitly to Week 3's process/signal knowledge — terminationGracePeriodSeconds is literally "how long between SIGTERM and SIGKILL," which you already know cold.	Deploy a pod with a custom grace period, force a delete, watch the signal timing with kubectl events and compare to your Week 3, Day 16 experiment.
66	Pod networking and Services mapped to Week 5 — Services are essentially load-balanced DNS + iptables/IPVS rules under the hood.	Inspect iptables/ipvsadm rules generated by a Service on your local cluster (if accessible) to see the Week 5 firewall knowledge made concrete.
67	ConfigMaps/Secrets mapped to Week 8's environment variable/config precedence knowledge — same concept, Kubernetes-native delivery mechanism.	Mount a ConfigMap as both an env var and a file in the same pod, observe both, connect to Day 50's precedence exercise.
68	Resource requests/limits mapped directly to Week 9's cgroups exercise — same kernel mechanism, YAML instead of CLI flags.	Set a pod memory limit too low on purpose, observe the OOMKilled status, cross-reference with your Week 9, Day 62 container experiment — same failure, two layers of abstraction.
69	Cloud VM tie-in (conceptual, no account needed): walk through how AWS EC2 / Azure VM / GCP Compute Engine instances are, underneath, exactly the Linux you've learned — security groups = Week 5 firewalls, IAM roles = Week 6 sudo/permissions thinking, EBS/managed disks = Week 4 block devices.	Pick one cloud provider's free-tier docs (reading only, no account needed yet) and for each major concept (VPC, security group, IAM role, EBS volume), write one sentence translating it into the Linux term you already know.
70	Final synthesis day. No new material at all.	Write or talk through (with me) a complete narrative: "I deploy a containerized app from my laptop to a Kubernetes cluster running on cloud VMs" — using only the vocabulary and mental models from these 10 weeks, start to finish. If you can do this without hand-waving, you have reached comfortable intermediate.

How to know you've actually reached "comfortable intermediate"
You're there when, on any unfamiliar Linux box (a cloud VM, a CI runner, a Kubernetes node via kubectl exec), you can without panic: orient yourself in the filesystem, check what's running and why something stopped, read logs to find a root cause, check network/port issues, and write a short script to automate the fix — explaining why at each step rather than just what command to type. That's the actual bar for DevOps/Platform/Cloud roles, and this plan is built to land you there in 10 weeks at 2-4 hrs/day, with room to slow down on any week that needs it.
Notes on pacing
This is dense for total beginners, but matches your time budget. If any single week takes 1.5 weeks instead of 1, that's normal and fine — the daily structure is a sequence, not a deadline. The buffer day in Week 4 is intentional; feel free to mentally add more buffer days wherever you need them. The thing that would actually hurt your progress is rushing past a shaky checkpoint just to stay on schedule.
